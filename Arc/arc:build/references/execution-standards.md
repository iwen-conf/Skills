# 实施执行标准（arc:build）

## 1. 适用范围

本标准用于 `arc:build` 的实现阶段，约束“计划分解、编码交付、验证证据、交接材料”的最低质量线。  
默认目标：**小步可审阅、结果可验证、风险可回退**。

## 2. Definition of Done（DoD）

以下项全部满足，方可判定“实施完成”：

1. 目标与范围明确（`what`/`why`/`out-of-scope` 已写入计划）。
2. 关键改动已落地，且符合项目既有风格与约束。
3. 至少一类验证已执行（单测/集成/E2E/静态检查中的任一组合）。
4. 验证结果有证据（命令、输出、时间戳、失败项说明）。
5. 受影响模块、回归风险、回滚策略已记录。
6. 已输出可直接评审的交接摘要（变更点、风险、后续动作）。

## 3. 变更单元标准

- 单次提交/变更应保持“最小闭环”：一个可独立验证的目标。
- 大任务按能力边界切分（后端/前端/数据层/配置层），禁止一次性混改。
- 每个变更单元都应绑定：
  - 输入依据（上游方案/需求文档）；
  - 影响面清单（文件、模块、接口）；
  - 验证命令（至少 1 条）。

### 3.1 远端检查点要求

- `Medium` / `High` 风险改动开始前，应确认当前分支至少具备一种可恢复边界：已推送远端分支、可回滚快照，或明确记录的阻塞原因。
- 若仓库尚无安全远端且目标托管为 Gitea，优先联动浏览器自动化流程在该服务器上创建仓库，再把 SSH 地址回填到本地 `.git` remote 并推送当前分支；仓库可见性按用户要求或现有策略决定。
- 对“仅本地长时间施工、没有远端检查点”的情况，必须在计划与交接中明确标红，不得默认忽略。

## 4. 验证分层要求

### 4.1 基础层（必需）

- 代码可构建或可运行。
- 至少完成一条关键路径验证。
- 对失败验证项做“原因 + 风险 + 后续动作”说明。

### 4.2 增强层（推荐）

- 对核心逻辑补充回归测试。
- 对外接口变更补充 Contract Test 或兼容性说明。
- 对高风险改动补充性能/安全冒烟验证。

### 4.4 工具负载控制与防崩溃 (CPU Load & Crash Prevention)

- **防崩溃首要原则**：无论是使用 Python 还是 Node/Bun 编写自动化脚本，**绝对禁止**写出导致 CPU 负载瞬间拉满或内存耗尽的恶劣代码。
- **严禁无延迟的死循环/状态轮询（Busy Loop）**：在进行状态等待或轮询时，必须显式加入延迟（如 Node 的 `await new Promise(r => setTimeout(r, 1000))` 或 Python 的 `time.sleep(1)`），严禁 `while(true)` 裸跑。
- **严禁失控的并发**：处理大批量文件或网络请求时，严禁无限制并行（例如直接把上万个 Promise 丢给 `Promise.all`，或随意开启数百个多进程/多线程）。必须实现批次处理（Chunking）或设定并发上限。
- **严禁使用脚本手写深层目录遍历**：在查找文件或扫描时，必须忽略 `node_modules`、`.git`、`.venv` 等庞大黑洞目录。凡是能用 `rg` (ripgrep) 或 `fd` 解决的查询，**绝对禁止**手写 Node/Python 脚本去递归扫盘。
- **使用轻量级工具替代脚本拼接**：优先使用经过极端性能优化的底层系统工具（`rg`、`fd`、`jq`、`awk` 等）来完成扫描、过滤和分析，而不是为了处理一点文本就去写一个重型的 Node 或 Python 脚本。

### 4.5 CLI-First 工具阶梯

- 默认按下面顺序选工具，而不是先写 Python：
  1. 搜索与读取：`rg`、`rg --files`、`fd`、`git diff`、`git status`、`find`、`sed`
  2. 过滤与聚合：`awk`、`sort`、`uniq`、`cut`、`xargs`
  3. JSON/接口结果处理：`jq`
  4. 浏览器自动化：`mcp_chrome-devtools_*` 工具（用于单点验证/原生集成）或 `agent-browser`（用于复杂流程/多步交互）
  5. 平台/仓库协作：`gh`、`just`
  6. 高性能结构化逻辑/并发处理：**Go** 语言（若需要编写一次性的高并发扫盘、复杂 AST 解析或 CPU 密集型任务，直接用 `go run` 运行单文件脚本，性能远超 Node/Python，且不会出现死锁吃满内存的问题）
  7. 轻量结构化逻辑：JS/Node，优先 `bun`（用于处理网络请求、JSON 转换等轻量 I/O 密集型任务）
  8. 会话管理：`tmux`
  9. 仓库原生脚本：项目已经提供的脚本优先直接复用
  10. Python：仅作最后兜底
- 常见替代策略：
  - 仓库找文件/找文本：优先 `rg` / `fd`，不要写 Python 遍历目录
  - 状态轮询/等待：优先 shell 循环或现成 CLI `wait` 能力，不要写 Python busy loop
  - 浏览器胶水：优先 `mcp_chrome-devtools_*` 工具或 `agent-browser` 视复杂程度而定，不要再包一层 Python
  - Git 信息采样：优先 `git status`、`git diff --stat`、`git log --oneline`
  - GitHub 操作：优先 `gh`，不要为了 PR / issue / repo 信息再开浏览器
  - 项目标准命令：若存在 `justfile`，优先 `just <task>`，避免重复拼命令
  - 长时任务：优先用 `tmux` 托管，而不是开 Python watcher

### 4.3 证据格式（建议）

```text
验证项: <name>
命令: <command>
结果: PASS/FAIL
证据: <stdout 摘要或报告路径>
时间: <ISO8601>
```

## 5. 风险控制与回滚

### 5.1 风险分级

| 级别 | 说明 | 处理要求 |
|------|------|----------|
| Low | 局部改动、低耦合 | 常规验证即可 |
| Medium | 多模块联动、存在兼容风险 | 增加回归验证与灰度建议 |
| High | 数据结构/关键链路/生产行为变更 | 必须给出明确回滚方案与人工确认点 |

### 5.2 回滚最小要求

- 记录可回滚边界（代码、配置、数据）。
- 给出回滚命令或操作步骤。
- 标记不可逆动作（如数据迁移）及补救策略。

## 6. 接口与契约变更要求

- 对外 API、事件、数据库 schema 变更必须显式记录。
- 破坏性变更需给兼容窗口或迁移方案。
- 若涉及版本策略，建议在交接中注明 `SemVer` 影响（major/minor/patch）。

## 7. 交接输出最低清单

交付时至少包含：

1. `implementation-plan.md`（目标、范围、分解、风险）。
2. `execution-report.md`（做了什么、为什么、如何验证）。
3. `change-summary.md`（评审速览版，含影响面与下一步）。
4. 验证记录（命令与结果，或报告链接）。

## 8. 常见反模式

- 只给“已完成”结论，不提供验证证据。
- 多个高风险点打包修改，无法定位失败根因。
- 接口已变更却未更新影响说明与调用方提示。
- 发现失败项后静默忽略，不写风险与后续动作。

## 9. 与其他技能协作

- 上游来自 `arc:decide`：优先消费其收敛方案与任务顺序。
- 下游联动 `arc:e2e`：涉及用户流程时补充真实路径验证。
- 发布前联动 `arc:gate`：需要 Go/No-Go 判定时输出门禁可消费证据。
