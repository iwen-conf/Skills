# 中国高校常见 UML / E-R 作图要求速记

本文不是为了复述整本教材，而是把高校课程、实验、课程设计、毕业设计中最常被老师检查的点，整理成可直接执行的建模约束。

## 1. 适用范围

当用户明确要求：

- “符合中国大学 / 高校 / 课程设计 / 毕业设计要求”
- “像课本 / 教材 / 答辩图那样画”
- “老师一眼能检查”

则优先执行本文规则；若老师或学院另有模板，以院系模板优先。

## 2. 通用规则

- 一张图只回答一个主问题，不要把多种语义混在一起。
- 线条尽量正交、少交叉、少回头。
- 节点名称简洁，避免把一整句需求写进图形。
- 同类元素对齐，层次清楚。
- 图中必须显式表达关键约束，不能只在口头说明里补充。
- 若流程涉及多个角色、部门、系统，优先用泳道、边界框或节点分区把责任划开。

## 3. 活动图

高校课程中，活动图最常被检查以下几点：

- 是否有初始节点。
- 是否有结束路径。
- 是否区分动作 / 活动、判定 / 合并、分叉 / 汇合。
- 是否把责任主体通过泳道分开。
- 是否把条件写在边上，而不是写进节点名字里。

执行规则：

- 起点必须画成实心圆。
- 终点优先使用活动终止或流程终止节点，不要只写“结束”文本。
- 动作节点使用圆角矩形，名称优先用“动词 + 宾语”。
- 判定与合并使用菱形；条件写在出边 guard 上，如 `[库存充足]`。
- 并发必须使用 fork / join 粗横条。
- 跨角色流程优先加泳道；泳道名应为角色、岗位、系统或部门，而不是动作。

## 4. 用例图

高校作业中常见扣分点是把“流程步骤”错画成“用例”。

执行规则：

- 参与者在系统边界外。
- 用例在系统边界内。
- 用例名称描述目标，如“提交订单”“查询成绩”，不要写“点击按钮”。
- `<<include>>` 表示公共必经能力，`<<extend>>` 表示条件性扩展。
- 参与者名写角色，不写具体某个人名。

## 5. 时序图

高校课程通常希望学生用时序图区分“谁先调用谁、什么时候返回、什么情况下分支”。

执行规则：

- 时间自上而下。
- 参与者从左到右排列，顺序稳定。
- 消息尽量区分同步调用、异步消息和返回。
- 关键条件分支用结构化片段或清晰分区表示。
- 不能只画主成功路径；关键失败、超时、重试至少要体现一条。

## 6. 类图

高校评图通常关注“类职责是否清楚、关系是否标准、是否乱塞字段”。

执行规则：

- 使用标准三段式类框：类名 / 属性 / 操作。
- 可见性用 `+`、`-`、`#`、`~`。
- 泛化、实现、依赖、关联、聚合、组合分别使用正确关系。
- 不要把数据库物理字段全量搬到类图中。
- 类名用名词，操作用动词短语。

## 7. 状态机图

高校常要求学生分清“状态”和“动作”。

执行规则：

- 一张状态机图只描述一个对象或核心业务实体的生命周期。
- 状态名应表示稳定状态，如“待支付”“已发货”，不要写成“提交订单”这种动作。
- 转换尽量写成 `事件 [条件] / 动作`。
- 初始状态、终止状态、关键异常迁移要清楚。

## 8. 组件图与部署图

这两类图在高校项目答辩中经常被要求成对出现，但语义容易混淆。

执行规则：

- 组件图描述逻辑模块、接口、依赖与职责边界。
- 部署图描述服务器、容器、节点、数据库、中间件与连接。
- 不要把逻辑组件直接画成物理节点。
- 如果系统依赖第三方平台、消息队列、对象存储或数据库，应在图中交代主要连接。

## 9. E-R 图

中国高校数据库课程最常见要求是：

- 使用陈氏画法。
- 区分实体、联系、属性。
- 标主键。
- 标基数。
- 不要把物理字段和外键直接画进去。

执行规则：

- 实体用矩形。
- 联系用菱形，名称用动词或动宾短语。
- 属性用椭圆；主键以下划线表示。
- 多值属性可用双椭圆，弱实体可用双矩形。
- 基数标在联系边靠近菱形的位置。
- 外键、字段类型、长度、索引、非空约束等物理信息不进入陈氏图。
