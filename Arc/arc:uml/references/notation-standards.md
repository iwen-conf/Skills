# 建模记法标准（UML + Chen E-R + drawio）

## 1. 默认源文件与交付要求

- 正式图源文件必须为原生 `.drawio`。
- 所有正式作图动作必须委托给 `drawio` skill 完成。
- 若用户要求导出格式，优先导出：
  - `.drawio.svg`
  - `.drawio.png`
  - `.drawio.pdf`
- 如果导出流程由 `drawio` skill 自动删除了中间 `.drawio` 文件，应把带嵌入 XML 的导出文件视为可编辑主产物，并在交付说明里写清楚。
- `arc:uml` 不再把 Mermaid / `.mmd` 作为默认交付路径；**时序图是明确例外**，可交付 `.mmd`，或在 `drawio` 中使用内嵌 Mermaid 语法。

## 2. UML 通用要求

- 语义基线默认对齐 `UML 2.5.1 / ISO 19505`。
- 图中元素命名必须能回溯到真实代码、配置、接口、角色、业务对象或数据库设计。
- 同一张图只服务一个主目标，不要混入过多视角。
- 关系类型不得混用：
  - 关联、依赖、聚合、组合、泛化要区分清楚。
  - 控制流、消息流、部署连接、数据关系不得互相冒充。
- 版式必须整洁可检查：
  - 连线方向清晰
  - 避免大量交叉
  - 同类元素对齐
  - 主线流程从左到右或从上到下保持一致

## 3. 中国高校场景下的通用审图点

下面这些要求在国内高校课程作业、实验报告、课程设计、毕业设计中最常见：

- 元素必须画全，不能只写说明文字代替图符号。
- 图名、节点名、参与者名要简洁明确，避免一段话式命名。
- 关键关系和约束必须显式画出，不能靠口头解释补。
- 一张图的语义边界必须清楚，例如：
  - 用例图不代替活动图
  - 活动图不代替时序图
  - E-R 图不代替物理表结构图
- 如果图跨角色、跨部门、跨系统，优先使用泳道、边界框或节点分区把责任划开。

详细图型规则请继续读取：

- [china-university-diagram-guidelines.md](./china-university-diagram-guidelines.md)

## 4. 活动图要求

- 必须有初始节点。
- 至少要有一条合法路径到达：
  - 活动终止节点，或
  - 流程终止节点。
- 动作或活动节点使用圆角矩形，名称优先使用动宾结构。
- 条件分支使用判定节点（菱形），条件写在控制流边上的 guard 中，如 `[校验通过]`。
- 分支汇回使用合并节点，不要省略。
- 并发必须使用 fork / join 粗横条，不得用菱形冒充并发。
- 跨角色流程优先使用泳道。
- 不要把数据库表、实现类或接口地址直接塞进活动节点名中。

## 5. 用例图要求

- 参与者位于系统边界外。
- 用例位于系统边界内。
- 用例名称描述“用户目标”而不是“实现步骤”。
- `<<include>>` 表示必经的公共行为复用。
- `<<extend>>` 表示条件性、可选性扩展。
- 泛化只在参与者或用例确有继承语义时使用。
- 不要把页面按钮、数据库表或接口路径直接画成用例。

## 6. 时序图要求

- **致命排版预警**：大模型生成原生 `drawio` XML 时序图时，因无法精确计算 `x/y` 绝对坐标，极易出现“所有连线从左侧原点堆积齐发、完全无视生命线”的排版崩坏。
- **强制约束**：针对时序图，**禁止**让大模型强行计算每一条消息连线的独立坐标。必须要求 `drawio` skill 使用内嵌的 Mermaid / PlantUML 文本节点（例如 `style="shape=mermaid;"` 的单元），或直接降级使用纯文本 Mermaid（`.mmd`）格式交付。
- **稳定承载建议**：如果必须交付 `.drawio`，优先把 `sequence.mmd` 包装成 draw.io 宿主文件，并保留嵌入的 Mermaid 源，而不是重新回到原生 XML 消息箭头。
- 时间方向默认自上而下。
- 生命线顺序稳定，避免同一参与者在图中多次改名。
- 消息要区分同步、异步、返回与自调用。
- 分支、可选和循环情形要通过 `alt` / `opt` / `loop` 等结构化片段表达。
- 两种或以上互斥情况必须使用 `alt` 组合片段矩形框，不要用多条箭头直接并排写“成功/失败”“通过/不通过”等分支结果。
- 单个条件成立才继续的可选场景使用 `opt` 矩形框；不要把条件说明直接塞进普通消息箭头里冒充分支。
- 关键异常、超时、重试或补偿流程不能全部省略。

### 布局约束（防面条图）
- **生命线排序**：按调用依赖从左到右排列，最大限度减少交叉线
  - 推荐顺序：前端/调用方 → Controller → Service → DAO/外部服务 → 基础设施
  - 被调用者放在调用者右侧，返回消息自然向左下方回指
- **消息数量限制**：单张图不超过 15-20 个消息，超限必须拆分子图或使用 `ref` 引用
- **激活条（Activation Bars）**：同步调用必须显示激活条（实心矩形），明确调用范围
- **间距控制**：生命线水平间距均匀（建议 160-200px），留出消息标签空间
- **分层表达**：复杂流程使用 `ref` fragment 引用子图，保持主图简洁

### 可读性检查清单
- [ ] 线条交叉数是否可控（每生命线交叉不超过 3 处）
- [ ] 消息标签是否清晰可读（无重叠、无挤压）
- [ ] 是否区分同步（实心箭头）、异步（开口箭头）、返回（虚线箭头）消息
- [ ] 两种不同情况是否放进同一个 `alt` 矩形组合片段，而不是靠箭头硬分叉
- [ ] 异常分支是否单独用 `alt` 片段表达，不混入主流程
- [ ] 生命线名称是否与代码类名/服务名一致，便于追溯

## 7. 类图要求

- 类使用标准三段式矩形：名称、属性、操作。
- 可见性使用标准记号：`+`、`-`、`#`、`~`。
- 只画当前讨论所需的关键属性和操作。
- 如果任务目标是领域建模，不要把整张表的所有物理字段都搬进类图中。
- 泛化、实现、聚合、组合、依赖必须分别使用正确关系。

## 8. 组件图与部署图要求

- 组件图表达逻辑模块、接口与依赖。
- 组件优先使用标准 component 记法或 `<<component>>` stereotype，不要退化成普通矩形框图。
- 组件之间的依赖优先使用标准依赖关系或显式接口记法，不要只把模块并排摆放却没有关系语义。
- 部署图表达物理节点、运行环境、制品与连接。
- 不要把“逻辑组件”和“物理节点”混在同一层语义里。
- 如果系统跨前端、后端、消息队列、数据库或第三方平台，至少要交代主要边界与连接方向。

### 部署图布局约束（防错位）

- **分区优先**：先按网络区、运行层或环境边界分区，再在分区内放节点。
  - 推荐顺序：入口/客户端区 → 接入层 → 应用层 → 中间件/数据层 → 外部平台。
- **结构先行**：优先先写结构化规格，再由脚本统一生成 draw.io XML，而不是让模型直接脑补坐标。
  - 推荐规格说明见 [deployment-layout-spec.md](./deployment-layout-spec.md)。
- **节点对齐**：同层节点顶边或中心线对齐，优先贴 10px 网格；不要凭感觉散点摆放。
- **容器嵌套**：机房、VPC、命名空间、集群、宿主机、容器等层级，优先用 container / swimlane 表达，而不是画成一堆孤立框。
- **连接绑定**：已知起点和终点的连接线，必须通过 `source` / `target` 绑定到节点 ID；不要用浮动 `sourcePoint` / `targetPoint` 硬拉线。
- **连线样式**：部署图连接线优先使用正交路由。
  - 推荐样式：`edgeStyle=orthogonalEdgeStyle;orthogonalLoop=1;jettySize=auto;html=1;`
- **少用手工拐点**：只有在自动正交路由确实无法表达时，才补 `Array as="points"` waypoints；不要把每条线都写成手工折线。
- **一图一主线**：如果跨区连接太多，拆成主部署图 + 局部子图，不要把所有网络通道塞进一张图。

### 部署图可读性检查清单

- [ ] 顶层节点是否按层级或分区整齐对齐
- [ ] 所有关键连接是否明确挂接到具体节点，而不是漂浮在画布上
- [ ] 连接线是否以正交为主，避免大量斜线和交叉
- [ ] 物理节点、运行制品、逻辑组件三种语义是否没有混层
- [ ] 是否清楚表达了用户入口、应用承载、数据存储和外部依赖

## 9. 状态机图要求

- 一张状态机图只描述一个核心对象或聚合根的生命周期。
- 状态名应描述稳定状态，不要把一次性动作写成状态。
- 转换尽量写清 `事件 [条件] / 动作`。
- 初始状态、终止状态、非法转换边界要明确。

## 10. E-R 图标准要求（陈氏画法）

输出 E-R 图时，必须基于真实的数据库表设计作为数据来源，并采用陈氏画法。陈氏图属于概念数据模型，不是物理表结构图。

### 必须画出的元素

- **实体（Entity）**：矩形框。
- **联系（Relationship）**：菱形框，名称用动词或动宾短语。
- **主键属性**：椭圆，属性名以下划线标识。
- **基数约束**：在联系边上标 `1`、`N`、`M` 等基数。

### 视情况增加的元素

- 普通属性：椭圆。
- 多值属性：双椭圆。
- 弱实体：双矩形。
- 弱联系：双菱形。
- 全参与：双线。
- 复合属性：父属性拆到子属性。

### 严禁出现的内容

- Crow's Foot / IDEF1X 符号。
- 外键字段作为属性椭圆直接画在实体上。
- 数据类型、长度、非空约束、索引、触发器等物理实现信息。
- 把类图或表结构图冒充陈氏 E-R 图。

## 11. 常见错误清单

- 图里只有框和箭头，没有记法语义。
- 用文字说明代替图元素。
- 一个图里同时混入静态结构、控制流和部署关系。
- 节点命名与项目术语完全对不上。
- E-R 图用“字段堆砌”代替实体-联系表达。
- 仍按 Mermaid 习惯思考布局，导致 draw.io 图缺少边界、对齐和可检查性。
