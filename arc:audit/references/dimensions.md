# 企业级软件项目评估 — 七维度框架

> 提炼自《企业级软件项目评估与专家能力进阶白皮书》，参考 ISO/IEC 25010 软件质量模型、TOGAF 企业架构框架以及现代软件工程最佳实践。

## 概述

评估一个软件系统，需既看"增量"（业务价值与架构设计），也看"存量"（技术债与维护成本）。  
在七维度总分之外，**必须额外产出两项专项分**：
- **业务成熟度分（0-10）**：衡量业务链路打通率、断链率与运营闭环成熟度
- **依赖健康分（0-10）**：衡量依赖过时率、漏洞风险与维护状态

## 维度 1: 架构设计与技术演进性

**Architecture & Technical Scalability**

决定软件生命周期长度的骨架。

### 检查项

| 检查项 | 评估要点 | 代码证据来源 |
|--------|---------|-------------|
| 领域边界与解耦 | 是否划分了清晰的领域边界？业务逻辑是否与基础设施解耦？是否存在过度耦合的"焦油坑"？ | 目录结构、import 图谱、domain/usecase/interface 分层 |
| 扩展性与弹性 | 是否支持水平/垂直扩展？是否具备限流、熔断、降级、背压？ | 配置文件、中间件代码、重试/超时逻辑 |
| 技术栈合理性 | 技术栈是否符合团队维护能力？是否有活跃生态？是否切合业务场景？ | go.mod/package.json/Cargo.toml、依赖版本 |

## 维度 2: 安全合规与权限管控

**Security, Compliance & Access Control**

企业级应用的底线。

### 检查项

| 检查项 | 评估要点 | 代码证据来源 |
|--------|---------|-------------|
| 身份认证与授权 | 是否有健壮的细粒度权限控制？是否合理采用 RBAC/ABAC/ReBAC？越权访问是否绝对隔离？ | auth middleware、casbin policy、JWT 签发逻辑 |
| 数据安全与隐私 | 传输加密(TLS)？存储加密？是否符合 GDPR/数据安全法？ | TLS 配置、数据库加密字段、敏感数据处理 |
| 防注入与漏洞防御 | OWASP Top 10 防御：预编译 SQL、CSP 策略、CSRF Token？ | SQL 查询构建方式、HTTP header 设置、表单处理 |

## 维度 3: 代码质量与工程规范

**Code Quality & Engineering Excellence**

代码是基石，直接决定可维护性与协作效率。

### 检查项

| 检查项 | 评估要点 | 代码证据来源 |
|--------|---------|-------------|
| 可测试性与覆盖率 | 是否有自动化测试金字塔？单元/集成/基准/模糊测试？Mock 隔离？ | *_test.go、*.test.ts、coverage 报告、fuzz 目录 |
| 静态分析与规范 | 是否有统一风格？CI 是否强制扫描（SonarQube/ESLint/GolangCI-Lint）？圈复杂度？ | .eslintrc、golangci.yml、CI 配置 |
| 文档与元数据 | API 文档(Swagger)？ER 图？C4 架构图？部署指南？是否与代码同步？ | swagger.yaml、docs/ 目录、CLAUDE.md |

## 维度 4: 业务价值与需求契合度

**Business Value & Product-Market Fit**

技术最终服务于业务。

### 检查项

| 检查项 | 评估要点 | 代码证据来源 |
|--------|---------|-------------|
| 目标对齐 | 功能是否精准满足核心业务需求？ | 需求文档、路由/API 与业务流对应关系 |
| 业务链路打通率 | 核心业务流（注册→下单→支付→履约→结算等）中，已实现并可端到端跑通的比例是多少？ | 路由/服务编排、E2E 用例、集成测试、事件流日志 |
| 业务断链与人工兜底 | 目前有哪些断链节点（手工补单、线下对账、人工审批）？断链是否可观测并可追踪？ | TODO/HACK、补偿任务、人工脚本、告警与工单记录 |
| 用户体验 | 交互链路是否合理？视觉层次是否清晰？是否有加载态/骨架屏/优雅错误提示？ | 前端组件、loading/error 处理、UX 流程 |
| 业务规则可演进性 | 关键规则是配置化/规则引擎化，还是散落硬编码？改规则是否需要频繁改代码与回归全链路？ | feature flag、配置中心、规则模块、硬编码规则 |
| ROI 与交付周期 | 开发成本与业务回报是否成正比？是否能快速交付 MVP？ | 功能完成度、迭代计划 |

### 业务成熟度专项评分（0-10）

> 该分数用于回答“打通了多少业务、还有多少业务断开”。

| 子项 | 权重 | 计算方式（建议） |
|------|------|----------------|
| 核心链路打通率 | 0.40 | `已打通核心链路数 / 核心链路总数` |
| 断链控制能力 | 0.25 | `1 - 断链节点数 / 关键节点总数`（含人工兜底惩罚） |
| 运营闭环完整度 | 0.20 | 需求→交付→监控→告警→复盘是否闭环 |
| 业务规则演进性 | 0.15 | 配置化比例、规则变更 lead time、回归成本 |

建议输出：
- 业务成熟度总分（0-10）
- 已打通链路清单、未打通链路清单、断链节点 Top N
- 每条结论附 `file:line` 或任务系统证据

## 维度 5: 运维可观测性与部署交付

**Observability & DevOps**

系统上线只是开始。

### 检查项

| 检查项 | 评估要点 | 代码证据来源 |
|--------|---------|-------------|
| 可观测性三支柱 | 结构化日志？分布式链路追踪(OpenTelemetry)？指标监控(Planning+Grafana)？告警？ | 日志库配置、tracing 中间件、metrics 端点 |
| CI/CD 流水线 | 是否全自动化？滚动更新/蓝绿/金丝雀？ | .github/workflows、Dockerfile、Makefile |
| 资源调度 | 容器化(Docker/K8s)？资源调度是否最优？是否有单点瓶颈或严重浪费？ | docker-compose.yml、k8s manifests、资源配额 |

## 维度 6: 团队效能与协作

**Team Capability & Collaboration**

团队健康度决定项目未来风险。

### 检查项

| 检查项 | 评估要点 | 代码证据来源 |
|--------|---------|-------------|
| 知识共享与 Bus Factor | 是否存在"只有一个人懂"的模块？是否有 Code Review 文化？是否有 ADR 沉淀？ | git blame 分布、PR review 记录、docs/adr/ |
| 敏捷与响应力 | 对需求变更的响应速度？迭代节奏是否稳定？复盘机制是否有效？ | commit 频率、issue tracker、CHANGELOG |

## 维度 7: 技术债务与演进阻力

**Technical Debt & Evolution Friction**

评估系统的"沉重程度"及未来迭代的"利息"成本。

### 检查项

| 检查项 | 评估要点 | 代码证据来源 |
|--------|---------|-------------|
| 依赖腐化与供应链风险 | 是否绑定了已停止维护的库？语言运行环境是否长期未升级？是否有"依赖地狱"？ | go.mod/package-lock.json 版本、dependabot 配置 |
| 过时依赖率 | 生产依赖中落后最新稳定版 ≥ N 个小版本的比例？关键基础库是否长期滞后？ | lockfile、`npm/pnpm/yarn outdated`、`go list -m -u` |
| 漏洞依赖暴露面 | 已知漏洞（CVE/GHSA）按严重级别分布如何？是否存在未修复高危漏洞？ | `npm audit`/`pip-audit`/`cargo audit`/`govulncheck` 报告 |
| 停维护依赖占比 | 是否使用 archived/abandoned 包？是否存在单点维护者依赖？ | 仓库活跃度、最近发布时间、维护者公告 |
| 过度设计与过度抽象 | 是否在简单 CRUD 中引入了过量 DTO 转换/接口/DI？架构是否失去"显式性"？ | 目录层级深度、interface 数量 vs 实现数量 |
| 硬编码与祖传逻辑 | 是否有魔法数字？是否有无注释的写死业务规则？ | 常量定义、配置 vs 硬编码比例 |
| 债务管理机制 | 团队是否将技术债纳入 Backlog？是否在每个 Sprint 预留重构资源(~20%)？ | issue tracker、TODO/FIXME/HACK 注释统计 |

### 依赖健康专项评分（0-10）

| 子项 | 权重 | 计算方式（建议） |
|------|------|----------------|
| 过时依赖率得分 | 0.35 | `10 × (1 - 过时依赖率)`，并对关键依赖额外扣分 |
| 漏洞风险得分 | 0.35 | 按 Critical/High/Medium/Low 漏洞加权扣分 |
| 停维护风险得分 | 0.20 | 停维护依赖占比越高，得分越低 |
| 升级自动化得分 | 0.10 | Dependabot/Renovate/定期升级流程覆盖度 |

建议输出：
- 依赖健康总分（0-10）
- 过时依赖 Top N（当前版本/目标版本/影响模块）
- 高危漏洞与停维护依赖清单（含修复优先级）

## 评分标准

每个维度采用 0-10 分制：

| 分数区间 | 评级 | 含义 |
|---------|------|------|
| 9-10 | 卓越 | 行业标杆级别，值得其他项目参考 |
| 7-8 | 良好 | 符合企业级标准，有少量改进空间 |
| 5-6 | 合格 | 基本可用，但存在明显短板需要关注 |
| 3-4 | 警告 | 存在较严重问题，影响长期维护和演进 |
| 0-2 | 危险 | 存在系统性缺陷，需要立即干预 |

## 专家能力参考

评估者应具备以下核心能力：

1. **跨范式编程语言与架构理解力** — 不受语法局限，判断设计模式与语言生态的契合度
2. **复杂业务的领域建模与解耦能力** — 识别鉴权逻辑侵入、熟悉 ABAC/ReBAC 高级模型
3. **全链路的工程质量防线构建** — 多维度测试生态、持续重构闭环（绞杀者模式）
4. **穿透底层的网络与基础设施视野** — 协议层选型（gRPC/WebSocket/KCP）、K8s 架构耦合
5. **商业 ROI 与技术妥协的权衡艺术** — 反对过早优化、技术债金融化管理
