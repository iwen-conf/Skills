# Evidence Checklist

## 1. Source Priority

按下面的顺序取信，越靠前越权威：

1. 运行时代码：后端控制器、服务、实体、路由、前端页面、菜单配置。
2. 数据库脚本：`*.sql` 中的 `CREATE TABLE`、初始化数据、字段注释。
3. 项目内正式设计文档：接口文档、业务流、模块说明。
4. 同学论文、旧版 Word、外部模板：仅提供章节名、表格样式和表述颗粒度。

禁止反过来做：先按模板写，再去源码里“找差不多的证据补齐”。

## 2. Deliverable Matrix

| 文档 | 必备证据 | 常见误区 |
|---|---|---|
| 系统用例规约 | 角色、登录/权限、核心业务控制器、前端页面入口 | 根据项目名脑补业务流程；照搬别人论文里的参与者 |
| 数据库物理设计三线表 | SQL 表定义、字段注释、主键、默认值、非空约束 | 直接抄 README 表清单；表数与 SQL 不一致还继续沿用 |
| 功能模块实现主要代码 | 关键控制器/页面/服务实现片段、真实文件路径和行号 | 用伪代码代替真实代码；只写文件名不写实现位置 |
| 测试用例设计 | 现有测试文件、接口行为、页面交互、业务规则 | 把未执行的结果写成“测试通过”；把测试设计写成运行截图报告 |

## 3. Recommended Evidence Scan

- 读取项目根目录规范文件：`AGENTS.md`、`CLAUDE.md`、`tasks/todo.md`
- 确认后端技术入口：`pom.xml` / `package.json` / `go.mod`
- 定位角色与登录：
  - `*Controller.java` 中的 `login` / `register` / `session`
  - Token、JWT、拦截器、权限判断逻辑
- 定位核心业务：
  - 商品/课程/资源等主实体控制器
  - 购物车、订单、评论、收藏、地址、消息等辅助控制器
  - 前端对应页面、路由和按钮行为
- 定位数据库：
  - `db/*.sql`
  - 建表语句里的字段、注释、主键、默认值
- 定位测试：
  - `src/test/`
  - `tests/`
  - Playwright、JUnit、pytest 等配置或实际用例

## 4. Word Delivery Conventions

- 默认使用 `python-docx`。
- 三线表要求：
  - 表头上框线
  - 表头下框线
  - 表尾下框线
- 页面默认 A4；代码较宽时允许横向 A4。
- 文件名优先使用“项目名 + 文档名”；只有在仓库中找到学号/姓名等硬证据时才切换到学生命名格式。

## 5. Render Verification Loop

优先使用下面的闭环：

1. `python-docx` 生成 `docx`
2. `soffice --headless --convert-to pdf`
3. `pdfinfo` 检查页型是否为 A4
4. `pdftotext -layout` 抽检：
   - 标题是否正确
   - 表格是否断裂
   - 代码块是否错位

如果环境无法渲染 PDF，必须在最终说明里明确写出“已完成文本生成，但未完成版式渲染校验”。
