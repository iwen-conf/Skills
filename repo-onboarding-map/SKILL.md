# repo-onboarding-map

## Overview

在最短时间建立代码库“可行动认知”，降低上手和改动风险。

**核心原则：先建立地图，再动代码。**

## Announce

开始时明确说明：  
“我正在使用 `repo-onboarding-map`，先建立仓库上手地图与安全改动边界。”

## When to Use

- 新成员接手项目
- 跨团队协作前需要统一认知
- 大规模改动前做入口摸底

## Input Arguments

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `repo_path` | string | 是 | 代码库根目录 |
| `target_area` | string[] | 否 | 优先关注模块/目录 |
| `task_goal` | string | 否 | 当前待实现任务目标 |

## Dependencies

- **编排契约**（推荐）：遵循 `docs/orchestration-contract.md`，通过运行时适配层分发扫描任务。

## Outputs

- 系统入口与启动路径
- 模块职责与边界图
- 常用命令与本地运行路径
- 变更落点建议（改哪、避哪）
- 风险提醒（高耦合区、历史雷区、慎改点）
- 快速上手路径（30-60 分钟可执行）

## The Iron Law

```
NO LARGE CHANGE WITHOUT A REPO MAP
```

未完成入口、边界和落点定位前，不得进行跨模块重构。

## Workflow

### Phase 1: 入口识别
1. 定位启动入口、构建入口、测试入口
2. 提取关键命令与环境依赖
3. 标注本地运行最短路径

### Phase 2: 边界建模
1. 划分模块职责（入口层/业务层/数据层/基础设施层）
2. 识别依赖方向与高耦合区域
3. 标注核心链路与脆弱点

### Phase 3: 改动策略
1. 输出“优先改动落点”
2. 输出“慎改区/禁改区”
3. 给出变更前检查建议（回归点/风险点）

### Phase 4: 上手交付
1. 生成上手地图（入口+边界+命令+落点）
2. 输出常见陷阱和排错起点
3. 给出下一步执行建议

## Quality Gates

- 输出必须包含“入口、边界、命令、落点”四项
- 改动建议必须给出依据（文件/模块）
- 对高风险区域必须给出回退建议

## Red Flags（必须停下重做）

- 未识别入口就开始改核心逻辑
- 只看目录名，不验证实际调用链
- 未区分“改动落点”与“影响范围”
- 跨模块改动无回归点定义
