### 一、 按照架构与范围分类（测试金字塔）

这类测试通常随着软件生命周期自底向上进行：

* **单元测试 (Unit Testing)：**
测试粒度最小，针对代码中的独立单元（如单个函数、方法或类）进行验证，确保基础逻辑正确。
* **集成测试 (Integration Testing)：**
将多个经过单元测试的模块组合在一起进行测试，重点验证模块之间的接口和交互是否正常。
* **端到端测试 (E2E Testing)：** 从用户的视角出发，模拟真实用户的操作路径，贯穿整个系统架构（从前端 UI
到后端数据库）进行验证。
* **验收测试 (Acceptance Testing)：** 通常由产品经理或客户参与（如 UAT
用户验收测试），纯粹从业务需求出发，确认软件是否达到了交付标准。

### 二、 按照测试是否可见代码分类

* **黑盒测试 (Black Box Testing)：** 把系统当成一个打不开的黑盒子，完全不考虑内部代码结构，只验证输入和输出是否符
合预期。**边界值测试**和**等价类划分**都属于这种方法的具体策略。
* **白盒测试 (White Box Testing)：** 直接针对代码内部逻辑进行测试，关注代码分支覆盖率、路径覆盖率等。
* **灰盒测试 (Gray Box Testing)：**
介于两者之间，既关注输入输出，也结合对系统内部架构和接口的了解来进行测试（例如 API 接口测试）。

### 三、 针对特定目标的非功能性测试

除了验证“功能对不对”，还有大量测试用来验证软件“好不好用”和“稳不稳定”：

**1. 性能与高可用测试**

* **负载测试 (Load Testing)：** 模拟预期的最大用户并发量，观察系统在正常高负载下的表现。
* **压力测试 (Stress Testing)：**
不断向系统加压，直到系统崩溃，目的是寻找系统的极限瓶颈在哪，以及崩溃后的恢复能力。
* **混沌测试 (Chaos Engineering)：** 故意在系统中随机制造故障（比如杀掉进程、拔掉网线、拉高
CPU），以验证分布式架构的容错和自愈能力。

**2. 稳定性与质量保障**

* **回归测试 (Regression Testing)：** 在修复 Bug
或增加新功能后，重新运行以前的测试用例，确保改动没有“破坏”原有的正常功能。
* **冒烟测试 (Smoke Testing)：** 来源于硬件测试（一通电没冒烟才继续测）。在软件中指拿到新版本后，先走一遍最核心的
基础主流程。如果冒烟不通过，直接打回重做，拒绝后续深入测试。
* **变异测试 (Mutation Testing)：** 测试你的“测试用例”质量。通过工具故意把你的业务代码改错（比如把 `+` 改成
`-`），看看你的单元测试能不能抓住这个错误。

**3. 安全与可用性**

* **安全/渗透测试 (Penetration Testing)：** 模拟黑客攻击，寻找系统的 SQL 注入、跨站脚本（XSS）、越权等安全漏洞。
* **兼容性测试 (Compatibility Testing)：** 确保软件在不同的浏览器、操作系统、分辨率或移动设备上都能正常工作。
